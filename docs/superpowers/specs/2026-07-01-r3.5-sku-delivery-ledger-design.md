# R3.5 — SKU Delivery Ledger design

**Date:** 2026-07-01
**Status:** approved (design), pending plan
**Slice:** R3.5 (Juan's post-R3 smoke asks #3 + #4).
**Builds on (shipped):** R1 (`skuContentOz`), R2 (`vendor_price_history`, `lib/admin/cost.ts`, `SkuCostPanel`), R3 (`vendor_delivery_items`, `loadDeliveryDetail`).

> A read surface over R3's deliveries: each SKU keeps its **delivery history** (linking to the
> delivery log) and a running **received total** in **$** and **oz**. No new data — it composes
> `vendor_delivery_items` + R2 prices + R1 `content_oz`. This is **received-to-date**, not true
> on-hand (R4 refines to received − counted). **No migration.**

---

## 1. The model in one breath

> For a SKU, walk its `vendor_delivery_items`: **$ received** = Σ `qty_received × (line unit_price
> ?? current SKU price)`; **oz received** = Σ `qty_received × content_oz`. Lines that can't resolve
> a price / an oz are **excluded and counted** (honest under-count). The per-delivery rows link to a
> new `/operations/receiving/[id]` delivery-detail page. Shown in R2's `SkuCostPanel` (AGM+).

## 2. Decisions (resolved with Juan)

- **$ basis:** line `unit_price` where present, else the SKU's current price (`loadCurrentSkuPrices`,
  R2). Lines with neither → excluded; surface `unpricedLineCount`.
- **oz basis:** `qty_received × skuContentOz(sku, measures)` (R1). Lines where `content_oz` is null
  (count/volume SKU with no `avg_oz_per_each` yet) → excluded; surface `missingOzLineCount`.
- **Delivery link:** dedicated **`/operations/receiving/[id]`** page (reuses `loadDeliveryDetail`);
  R3's recent-deliveries list is also wired to link there.
- **Placement:** extend R2 `SkuCostPanel` with a "Deliveries" section. **Authority AGM+ (≥6)** for
  the ledger; the delivery-detail page is **KH+ (≥4)** view (receivers + managers), IDOR-bound.
- **No migration** — pure read/aggregation over existing tables.

## 3. Data layer (`lib/admin/cost.ts` — extends R2)

- **`SkuReceivingLedger`** = `{ receivedDollars: number; receivedOz: number; unpricedLineCount: number;
  missingOzLineCount: number; deliveries: Array<{ deliveryId: string; date: string; vendorName: string;
  qty: number; unitPrice: number | null }> }`.
- **`loadSkuReceivingLedger(actor, skuIds): Map<string, SkuReceivingLedger>`** (AGM+, `COST_READ_MIN`):
  1. Load `vendor_delivery_items` for `skuIds` (join `vendor_deliveries` for date + vendor_id;
     numeric→`Number()` coercion on qty/unit_price).
  2. `currentPrices = loadCurrentSkuPrices(skuIds)`; `contentOzById` via `skuContentOz` over the SKUs'
     pack inputs (load `vendor_items` pack fields + `measure_units` — mirror how the admin view builds
     `contentOzById`).
  3. Per line: `dollar = qty × (unitPrice ?? currentPrices[sku] ?? null)`; if null → `unpricedLineCount++`,
     contributes 0. `oz = qty × (contentOz[sku] ?? null)`; if null → `missingOzLineCount++`, contributes 0.
  4. Aggregate `receivedDollars`/`receivedOz` + collect the per-delivery rows (most-recent first),
     hydrating vendor names in a batch.
- Empty for SKUs with no deliveries (returns a zeroed ledger or omit — the panel renders "no deliveries").

## 4. Delivery-detail page

- **`app/(authed)/operations/receiving/[id]/page.tsx`** — `requireSessionFromHeaders`, KH+ (≥4) or
  redirect; calls `loadDeliveryDetail(auth, id)` (already IDOR-binds the delivery's location → 404 as a
  redirect for a page). Renders header (vendor, date, invoice #/total, notes, received-by) + the lines
  (SKU name, qty, unit price, observed oz, notes). `DashboardBackLink`.
- **Wire links:** R3's recent-deliveries list (`/operations/receiving`) entries + the SKU ledger's
  delivery rows both `Link` to `/operations/receiving/[id]`.

## 5. `SkuCostPanel` extension

Add a **Deliveries** section under the existing cost/price content:
- **Received totals:** "Received: $X · ≈ N oz" + a muted exclusion note when
  `unpricedLineCount`/`missingOzLineCount` > 0 ("3 unpriced · 1 without oz").
- **History:** up to ~5 recent delivery rows (date · vendor · qty), each a `Link` to the delivery page.
- Threaded via a new `ledger: SkuReceivingLedger` prop, loaded on both SKU pages (`/admin/skus` +
  vendor-detail) alongside the existing `skuCost` — `loadSkuReceivingLedger` over the page's SKU ids.

## 6. Verification

- `npx tsc --noEmit` + `npm run build` clean.
- Throwaway tsx smoke (deleted before commit): `loadSkuReceivingLedger` over the one live SKU that has a
  delivery → asserts `receivedDollars`/`receivedOz` are numbers, `deliveries.length ≥ 1`, and the
  exclusion counts behave (a priced+content line contributes; an unpriced or no-content line increments
  the right counter). numeric→number coercion asserted.
- **Operator smoke (Juan, preview):** operator flows unchanged; on `/admin/skus`, a SKU that received a
  delivery shows its received $ + oz totals + a delivery-history entry that links to the delivery page;
  the receiving recent-list entries now open the same delivery page.

## 7. Boundary

- **Received-to-date only.** True on-hand ("how much we have now" = received − counted) = R4.
- No migration. No new write path (read + navigation only).
- Ordering / R3b email-capture unaffected.

## 8. Open decisions deferred
- **D-decay:** received totals are lifetime cumulative in R3.5; a date-window ("received this month")
  can come with R4/R5 reporting if wanted.
- **D-detail-auth:** the delivery-detail page is KH+ view; if managers want an admin-tree copy under
  `/admin`, that's a trivial later add (same loader).
