## 2. The architectural model worth understanding before touching anything

### 2.1 Three-checkpoint operational loop

CO closing operations form a continuous quality loop, not three independent checkpoints:

```
Closing → Opening → Mid-day Prep → (next) Closing
```

Each checkpoint's quality affects the next. Closer's prep estimates affect the opener's verification work. The opener's quality affects whether mid-day prep is needed. Mid-day prep keeps the closing tight. The system reinforces this loop by tracking quality data per step. Build #1 ships the Closing checkpoint; Build #2 ships Prep; Build #3 ships Opening. Synthesis View (Build #5) is where the loop becomes legible to management.

### 2.2 Three artifact types in the loop

- **Closing instance** — has TWO phases:
  - **Phase 1 (shipped in Build #1):** cleanliness checklist + 50 items + role-leveled completion + Walk-Out Verification gate
  - **Phase 2 (lands in Build #2):** AM Prep List generation (closer estimates tomorrow's prep needs at end of shift)
  - One PIN attestation covers both phases. The closing UI's Review section has a function-prop seam (`onContinue`) that opens PinConfirmModal in Build #1; in Build #2 it'll route to Phase 2 first, then PinConfirmModal. See SPEC_AMENDMENTS.md C.19.

- **Prep instance** — single artifact type with two trigger paths:
  - `triggered_by: 'closing'` — closer-initiated AM Prep List at end of shift
  - `triggered_by: 'manual'` — operator-initiated mid-day prep when someone notices depletion
  - Both produce `checklist_instances` rows where `template.type = 'prep'`, distinguished by a new `triggered_by` field. **No new `prep_instances` table** — reuses existing `checklist_*` infrastructure. See SPEC_AMENDMENTS.md C.18.

- **Opening instance** — verifies closing + validates AM Prep, **NOT** count-collection. The opener spot-checks the closer's AM Prep estimate and surfaces disagreements; counts are operator judgment captured at closing, not collected at opening. See SPEC_AMENDMENTS.md C.20.

### 2.3 Multi-author closing model (Model A)

Closing is multi-hour, multi-person. Items A, B, C, D get ticked throughout the shift — Crunchy Boi closes at 2pm, 3rd Party at 4pm, evening prep, etc. Items save individually for everyone via the optimistic UI flow in `ChecklistItem`. The closing instance persists across sessions until finalization (no auto-submission, no per-user submit).

**Finalize UI is gated by TWO conditions** (per SPEC_AMENDMENTS.md C.26):

1. `actor.level >= 4` — security gate; only KH+ can lock up the shop
2. `walkOutVerificationComplete` — operational gate; all 5 items in the "Walk-Out Verification" station must have live (non-superseded) completions. The "I'm the last out" signal.

Both must hold. The person who completes the 5th Walk-Out Verification item is the finalizer; finalize affordance unlocks for THAT person. Single PIN attestation per closing.

Three architectural layers in total:
- **Item completion** — open to actors whose level satisfies each item's `min_role_level` (lib + RLS)
- **Operational finalize-ready** — Walk-Out Verification station fully complete (UI gate)
- **Finalize attestation** — PIN re-entry by the finalizer, role-checked against highest completed `min_role_level` (lib)

## 7. Working rhythm established (read before first action)

This is how Juan and the prior session worked. Future-Claude should preserve the cadence — it's why Build #1 shipped clean.

1. **Surface design decisions before writing code.** Pattern from steps 4, 6, 9 — Claude Code surfaces alternatives + recommendation, Juan reviews, decisions lock, then implementation begins. The 12-dimension surface in step 9 (closing page) is the canonical example. Saves rework.

2. **Surface code before commit.** Review gate at the end of implementation, not after. Catches issues before they enter git history. Independent calls Claude made during writing get explicitly flagged with rationale; Juan accepts or pushes back; final commit reflects the converged understanding.

3. **Commit per step, PR per step.** Small focused PRs through the CI gate. No bundling. No admin bypass on the gate. The build sequence (steps 1–10) shipped as PRs #7–#19 — each one is reviewable in isolation.

4. **Reset to fresh `origin/main` between PRs from the same long-lived branch.** Run `git fetch origin main && git reset --hard origin/main` before starting the next step's work, then commit on top of the freshly-reset branch. Captured in [`AGENTS.md`](../AGENTS.md) durable knowledge — the "Squash-merged PRs from a long-lived branch require rebase-onto-main between PRs" entry. Skipping this caused PR #10 to come up `CONFLICTING` and need a rescue rebase.

5. **Push back on flawed assumptions in real-time.** Don't build on bad foundations. Fix the foundation first. Juan's operational pushback during step 8 (dashboard ≠ reports console — they coexist; future console deferred to a separate module), step 9 (Model A multi-author closing, NOT single-author), step 10 (Walk-Out Verification gate, not role-only; notes visibility tiers; tips role correction; trainer-as-tag) caught real architectural gaps before code shipped.

6. **Architecture follows operations, not spec.** When spec abstraction conflicts with how CO actually runs, capture the divergence in `docs/SPEC_AMENDMENTS.md` and build to operational reality. The 11 amendments captured in Build #1 are Juan's operational wisdom translated into durable architectural decisions. The spec is a starting reference, not a binding contract.
